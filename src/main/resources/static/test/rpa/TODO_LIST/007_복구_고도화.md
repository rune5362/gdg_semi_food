# 007 복구 고도화

## 목적
- 장시간 실행 중 상태와 외부 사이트 일시 오류에 대응할 수 있는 복구 체계를 준비한다.

## 1. 자동 복구
- `@Scheduled`를 사용해 장시간 `RUNNING` 상태인 데이터를 찾아 복구한다.
- `SemiApplication`에 `@EnableScheduling`이 추가되었다.
- `RpaRecoveryService.recoverStaleRunningLogsOnSchedule()`가 5분 간격으로 장시간 `RUNNING` 로그를 복구한다.
- 기본 복구 기준은 `startedAt`이 현재 시각보다 60분 이상 오래된 `RUNNING` 로그다.
- `/rpa/recover-stale?staleMinutes=60`으로 수동 복구를 실행할 수 있다.

## 2. 재시도 강화
- 외부 사이트의 일시적 오류에 대비해 3회 내외 재시도 로직을 둔다.
- `TrendKeywordParserService`의 네이버 키워드 조회에 3회 재시도를 추가했다.
- `SupplierAndProductService`의 네이버 공급자/상품 조회에 3회 재시도를 추가했다.
- 새 라이브러리 없이 간단한 반복문 기반 재시도로 구현했다.

## 3. 알림 확장
- 실패 알림은 메일 또는 슬랙 같은 관리자 채널로 확장 가능하게 설계한다.
- 현재 메일/슬랙 의존성이나 알림 서비스는 확인되지 않았다.

## 4. 이력 관리
- 알림 이력과 재처리 이력을 남길 수 있는 구조를 검토한다.

## 완료 기준
- 장시간 `RUNNING` 상태를 자동으로 탐지하고 복구할 수 있다.
- 실패와 재처리 흐름을 추적할 수 있는 확장 지점이 있다.
- 복구된 로그는 `FAILED`로 종료되고 `message`에 자동 복구 사유가 남는다.

## 추가 구현 필요
- 알림 서비스는 아직 구현되지 않았다.
- 재시도 간 대기 시간(backoff)은 아직 없다.
- 다중 서버 환경에서는 복구 스케줄러가 여러 인스턴스에서 동시에 실행될 수 있으므로 DB 기반 잠금 정책을 추가 검토해야 한다.

## 이번 단계 구현 결과
- `@EnableScheduling`을 추가했다.
- `RpaRecoveryService`를 추가했다.
- `RpaRecoveryResponse`를 추가했다.
- `RpaLogRepository.findAllByStatusAndStartedAtBefore()`를 추가했다.
- `/rpa/recover-stale` API를 추가했다.
- 네이버 `RestClient` 호출에 3회 재시도를 추가했다.
